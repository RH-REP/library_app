# Library Feature Brainstorm

Issue #4 の依頼に対して、図書館アプリに必要そうなものを採否未決の候補として整理した。

この文書は「今すぐ全部作る一覧」ではなく、「後で採用不採用と優先順位を決めるための候補集」である。

## Issue 本文で最初に挙がっていた候補

- タグ
- フォルダ構成
- 用語集
- PDF scan program
- html
- 原本の保存
- 原本のマスターテーブル

## 候補一覧

### 1. 整理軸と基本データ

- タグ
- フォルダまたはコレクション
- 用語集または統制語彙
- 原本のマスターテーブル
- 資料タイトル
- 著者
- 出版社
- 出版年
- ISBN や識別子
- 言語
- 種別
  例: 書籍、論文、PDF、Web メモ、スキャン資料
- 状態
  例: 未整理、取り込み済み、OCR 待ち、読書中、要確認、整理完了
- 関連リンク
  例: 同テーマ、参照元、続き物、引用関係

### 2. 原本保存と取り込み

- 原本ファイルの保存場所
- 原本ファイル名の命名規則
- 原本と正規化レコードを結ぶ ID
- PDF scan program
- 既存 PDF の取り込み導線
- OCR 結果の保存場所
- 原本は変更せず、派生データを別管理にする方針
- 取り込み用 inbox
  例: 未整理の PDF や画像を一時的に置く場所
- 重複検出
  例: hash、ファイル名、書誌情報の一致
- 原本の版管理
  例: 再スキャン版、修正版、圧縮版

### 3. 検索と参照

- キーワード検索
- タグ検索
- フォルダ/コレクション参照
- 著者、年、種別などのメタデータ絞り込み
- 用語集から関連資料へ飛ぶ導線
- 原本と要約、メモ、OCR テキストの相互リンク
- 関連資料の推薦
  例: 同タグ、同著者、同テーマ
- 保存済み検索条件
- 最近見た資料
- 未整理資料一覧

### 4. 読書メモと追加調査

- 資料ごとのメモ欄
- 要約
- 引用メモ
- 疑問点メモ
- 追加調査タスク
- 「あとで読む」「比較する」「再確認する」などのフラグ
- 用語集への追加候補
- 関連資料の収集依頼
- 読書ログ
  例: いつ読んだか、どこまで読んだか

### 5. Web/HTML 画面

- HTML ベースの一覧/詳細画面
- 左ペインのフォルダツリーまたはコレクション一覧
- 上部の検索ボックスとフィルター
- 右ペインの詳細ビュー
- PDF viewer
- 原本、OCR、メモの分割表示
- タグ編集 UI
- 原本登録フォーム
- マスターテーブル編集画面
- スマホでも読める最低限の responsive layout

### 6. 運用と境界

- `main_artifact/web_app/` と個別データの分離
- 原本は software 本体ではなく private data として扱う方針
- demo 用 fixture と実データの分離
- import/export の契約
- backup 方針
- データ更新履歴
- AI を使わなくても検索できる導線
- 後から別 UI や別保存方式へ移れるようにするための ID 設計

## 最小構成の叩き台

採否は未決だが、最初の MVP 候補としては次の組み合わせが考えやすい。

- 原本の保存場所を決める
- 原本のマスターテーブルを 1 つ作る
- タグと最低限のフォルダ/コレクションを持つ
- PDF と既存ファイルを登録できる
- タイトル、著者、タグで検索できる
- HTML で一覧と詳細を見られる
- 資料ごとに短いメモを残せる

## 後で決める論点

- タグとフォルダは両方必要か。片方を主、片方を補助にするか。
- 用語集は free-form メモで始めるか、統制語彙として運用するか。
- 原本のマスターテーブルは、資料単位、ファイル単位、版単位のどこを主キーにするか。
- PDF scan program はアプリ内機能にするか、外部ツール連携にするか。
- OCR は最初から入れるか、原本保存と登録だけで始めるか。
- HTML 画面は静的デモから始めるか、最初から Web app の土台に乗せるか。
- 原本、OCR、メモ、用語集をどうリンクするか。

## Follow-up: Alpha Development Scope

Issue #5 の依頼に合わせて、上の候補からアルファ開発で必須な要素だけを抽出する。

ここでのアルファは、「図書館アプリの中核が成立するかを確認できる最小開発段階」と定義する。重要なのは、実データに近い原本を登録し、AI なしで検索し、原本を参照できることである。

### アルファで必須にする要素

#### 1. 原本の保存場所

- 原本ファイルを置く場所を決める。
- software 本体と個別データを分ける。
- PDF を含む既存ファイルを登録対象にする。

理由:
- 「検索できる」だけでは足りず、最終的に原本へ戻れないと図書館として成立しないため。

#### 2. 原本のマスターテーブル

- 原本ごとの主キーを持つ。
- 原本パスまたは保存先参照を持つ。
- 原本と表示用メタデータを結びつける。

理由:
- 登録、検索、参照の全導線の中心になるため。

#### 3. 最小メタデータ

アルファでは次を最低限にする。

- タイトル
- 著者
- 種別
- タグ
- 状態
- 原本 ID

補足:
- 出版社、出版年、ISBN、言語などは有用だが、最初のアルファ完了条件からは外してよい。

#### 4. 非 AI 検索

- キーワード検索
- タグ検索
- 種別または状態での絞り込み

理由:
- `goal.md` の必須条件が「AI に頼らなくても検索できる導線」であるため。

#### 5. 一覧/詳細の Web/HTML 画面

- 検索入力
- 一覧表示
- 詳細表示
- 原本への遷移または参照導線

理由:
- アルファ段階で、ユーザーが実際に「探して開く」体験を確認できる必要があるため。

#### 6. 基本登録導線

- 原本を登録できる。
- マスターテーブルへ最小メタデータを追加できる。
- 登録したデータが検索と詳細表示へ反映される。

理由:
- 検索だけ先に作っても、データ投入の経路がなければ実運用へ繋がらないため。

### アルファでは必須にしない要素

次は重要だが、アルファ完了条件からは外す。

- フォルダ階層
- 用語集
- PDF scan program
- OCR 結果の保存と検索
- 関連資料推薦
- 保存済み検索条件
- 最近見た資料
- 読書ログ
- 高度なメモや追加調査タスク

判断理由:
- これらは「中核が成立した後の運用改善」に近く、先に入れるとアルファの境界が広がりすぎるため。

### タグ、フォルダ、用語集の扱い

アルファでは次のように絞るのが妥当である。

- タグ: 採用する
- フォルダ階層: 後回し
- 用語集: 後回し

理由:
- タグだけでも最初の分類・絞り込みは成立する。
- フォルダ階層は UI と保存ルールの設計を増やす。
- 用語集は運用ルールが未確定なまま入れると維持コストが高い。

### PDF の扱い

アルファでは「PDF を登録し、検索結果から開ける」までを必須にする。

アルファ完了条件に入れないもの:

- 紙からのスキャン
- OCR
- OCR テキスト検索

理由:
- PDF の原本保存と参照だけでも、最初の図書館体験は成立する。
- scan/OCR は I/O、精度、保存形式の設計負荷が高く、別マイルストーンに分けるべきである。

### アルファ開発マイルストーン

#### M1. データ境界と原本契約

作るもの:

- 原本保存場所のルール
- 原本マスターテーブルの最小 schema
- software と private data の接続ルール

完了条件:

- 原本をどこに置き、どの ID で管理するかが決まっている。
- 最小メタデータ項目が決まっている。

#### M2. 登録フロー

作るもの:

- 原本登録方法
- マスターテーブルへの新規追加方法
- タグと状態の初期運用

完了条件:

- 1 件以上の原本を追加できる。
- 追加した原本が一覧や検索対象に入る。

#### M3. 非 AI 検索

作るもの:

- キーワード検索
- タグ検索
- 種別または状態フィルター

完了条件:

- 登録済みデータから目的の資料を絞り込める。
- AI を使わずに検索導線が成立する。

#### M4. 一覧/詳細/原本参照 UI

作るもの:

- 一覧画面
- 詳細画面
- 原本参照導線

完了条件:

- 検索結果から対象資料の詳細へ進める。
- 詳細から原本を開く導線がある。

#### M5. アルファ検証

作るもの:

- 実際の想定データでの確認メモ
- 残課題一覧
- ベータ以降へ送る項目の整理

完了条件:

- 「登録できる」「検索できる」「原本へ戻れる」が一連で確認できる。
- OCR、用語集、フォルダ階層など後続課題が切り分けられている。

### アルファ完了の定義

次を満たせば、アルファとして十分である。

- 原本の保存場所が分かれている。
- 原本マスターテーブルがある。
- 最小メタデータ付きで原本を登録できる。
- AI なしで検索できる。
- 一覧/詳細から原本へ到達できる。
- PDF を最低 1 種類の代表原本として扱える。

逆に、次が未実装でもアルファ未完了とはしない。

- OCR
- scan program
- 用語集
- フォルダ階層
- 推薦
- 高度なメモ運用

## Follow-up: Alpha Source Sample Collection

Issue #5 の追加コメントに合わせて、alpha 開発用の原本サンプルを 3 種類の形式で保存した。

保存先:

- `main_artifact/fixtures/demo_programming_tech_library/source_samples/`
- `main_artifact/fixtures/demo_programming_tech_library/records/alpha_source_samples.json`

### 作成したサンプル

#### 1. HTML

- `source_samples/html/requirements_engineering_overview.html`
- 対応テーマ: 何を作るか明確にする
- 技術・分野: 要求工学・仕様化

用途:
- HTML 原本の登録確認
- HTML preview の確認
- 最小メタデータ付与の確認

#### 2. PDF

- `source_samples/pdf/software_design_principles.pdf`
- 対応テーマ: 変更しやすく設計する
- 技術・分野: ソフトウェア設計

用途:
- PDF 原本の登録確認
- PDF preview または原本参照導線の確認
- PDF を代表原本として扱う alpha 条件の確認

#### 3. Image

- `source_samples/image/software_quality_practices.png`
- 対応テーマ: 正しさを継続的に確認する
- 技術・分野: ソフトウェアテスト・品質保証

用途:
- 画像原本の登録確認
- 画像 preview の確認
- HTML/PDF 以外の原本形式に対する導線確認

### 対応表の保存

`records/alpha_source_samples.json` に、次を保存した。

- source type
- 対応する重要要素
- 技術・分野名
- 代表的な手法
- source path
- tags

これにより、後続の登録フロー設計や検索 UI 実装時に、最初の fixture index として使える。

### このサンプル収集の位置づけ

今回の 3 件は、外部配布物をそのまま集めたものではなく、共有可能な synthetic fixture として生成した。

この判断にした理由:

- 著作権や配布条件に依存しない。
- alpha で確認したい論点に合わせて内容を小さく調整できる。
- HTML、PDF、画像の 3 形式を安定して repo に保存できる。

### 次に使う場面

このサンプル群は、少なくとも次で使える。

- 原本マスターテーブルの first cut schema 検討
- 登録フローの入出力確認
- 検索 UI の fixture データ
- HTML/PDF/画像 preview の挙動確認
