# 制作目標

情報を収集・集約し、必要な情報を検索・参照・追加調査できる Web 系の情報管理システムを作る。AI 秘書による案内は補助機能として検討するが、AI に頼らなくても検索できる導線を必須にする。PDF スキャンや PDF 取り込みも扱えるようにしたいが、実装範囲と方法は確認後に決める。

## 背景

ユーザーは、散らばった情報を集め、あとからスムーズに探して参照できる仕組みを求めている。AI に案内させる可能性はある一方で、検索や参照の基本機能は AI なしでも成立させたいという希望がある。

当面の開発テーマは「プログラミングの技術」に関する図書館とし、ソフトウェアは Web app を想定する。現時点では、その個別データも private な開発履歴として commit して保存してよい。開発段階では配布データへの一時的なコンタミは重大論点にしないが、まず `main_artifact/web_app/` と `main_artifact/private_data/` を物理的に分けて扱えることを優先する。

その他の利用場面、対象データ、既存資料、避けたい作業は未記入のため、以下は初回 issue からの仮置きである。

## 成功条件

- 情報を登録・集約し、必要な情報を検索できる主要導線が整理されている。
- AI 秘書を使う場合でも、キーワード検索や絞り込みなどの非 AI 検索が利用できる。
- 検索結果から原情報を参照でき、必要に応じて追加調査につなげられる。
- PDF スキャンまたは PDF 取り込みを扱う方針が決まっている。
- 実運用で扱う個別データ、PDF、メモ、収集依頼が、ソフトウェア本体に混入しない境界が定義されている。
- private な開発中に commit した個別データを、配布用 archive または clean repository から除外できる。
- `main_artifact/web_app/`、`main_artifact/private_data/`、`main_artifact/fixtures/` の役割が分かれ、Web app と個別データの置き場が迷わない。
- 先に原案を作り、3 段階のレビューを挟む進行になっている。

## 作る範囲

- 情報収集・集約・検索・参照・追加調査の基本フローの整理。
- AI 秘書を補助として使う場合の位置づけの整理。
- AI に頼らない検索機能の要件整理。
- PDF スキャン、PDF 取り込み、OCR などの扱い方の検討。
- ソフトウェア本体、個人運用データ、サンプルデータの分離方針の整理。
- private 開発履歴と配布用成果物を分ける release boundary の整理。
- Web app 前提の初期フォルダ構成の整理。
- Web またはそれに準じる利用画面の原案作成。
- 3 段階レビューを含む制作工程の定義。

## 作らない範囲

- AI だけで検索・回答を完結させる設計。
- 本番運用、公開範囲、認証、権限、バックアップ方式の確定。
- 具体的な実装技術、AI モデル、OCR エンジン、データベースの確定。
- 大量データ移行や外部サービス連携の実装。
- 実際の個別データ、PDF、個人メモ、収集依頼をソフトウェア本体へ同梱すること。
- 配布用成果物の無汚染化を、現時点で自動化まで完了させること。
- ユーザーがまだ提示していない既存資料や業務要件の推測による確定。
