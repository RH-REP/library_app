# 工程概要

このプロジェクトは、最初に原案を作り、その後に 3 段階のレビューを挟んで、情報収集・検索・参照・追加調査・PDF 取り扱いを行うシステムの方向性を固める。初期段階では細かい実装を決めすぎず、ユーザーが次に判断しやすい単位へ分けて進める。

## 運用しながら開発するための境界

実運用で得た個別データは、ソフトウェア本体とは別の運用データとして扱う。ソフトウェア側に戻すのは、コード、schema、import/export 契約、画面要件、検索要件、架空または十分に匿名化した fixture に限定する。

実データから見つかった課題は、次の順に扱う。

1. private runtime data で問題を観察する。
2. 個別情報を削り、構造だけを issue にする。
3. 必要なら架空 fixture または匿名化 fixture で再現する。
4. software 側で汎用機能として実装する。
5. private project repo へ取り込み、実データで確認する。

## 工程1: 目標整理と原案化

初回 issue の内容から、制作目標、成功条件、作る範囲、作らない範囲、未確定事項を整理する。あわせて、最初の作業単位として `sub_artifact/001_artifact/` を初期化する。

成果物:

- `main_artifact/goal.md`
- `main_artifact/development_process.md`
- `sub_artifact/001_artifact/sub_goal.md`
- `sub_artifact/001_artifact/plan.md`
- `sub_artifact/001_artifact/work_log.md`
- `sub_artifact/001_artifact/artifact.md`
- `issue_log/001_artifact/issue_1_initial_setup.md`

完了条件:

- ユーザーが「何を作るか」を確認できる。
- 仮置きの内容と未確定事項が分かれている。
- 次の作業 issue を切れる状態になっている。

確認事項:

- この制作目標で進めてよいか。
- 最初に作るべき成果物は、要件定義、画面原案、技術検証、プロトタイプのどれか。
- PDF スキャンや AI 秘書をどの優先度で扱うか。

## レビュー1: 原案レビュー

工程1の内容をユーザーが確認し、以降の作業単位を決める。

確認事項:

- `goal.md` は worker が読んで迷わないか。
- AI に頼らない検索導線が必須条件として明示されているか。
- PDF スキャン、追加調査、AI 秘書の位置づけが大きすぎる仮定になっていないか。
- 未確定事項のうち、次工程前に必ず決めるべき項目はどれか。

終了条件:

- ユーザーが次の制作単位に進むことを了承している。

## 工程2: 要件と利用体験の整理

レビュー1で選ばれた方向に沿って、利用者、対象情報、登録方法、検索方法、参照方法、追加調査の流れを整理する。必要に応じて、画面遷移案や簡易ワイヤーフレームを作る。

成果物:

- 対象ユーザーと主要ユースケースの整理
- 情報登録・検索・参照・追加調査のフロー
- PDF 取り扱い方針の候補
- AI 秘書を使う場合と使わない場合の導線比較

完了条件:

- 最小構成で必要な機能が分かる。
- 後回しにする機能が分かる。
- 実装や技術検証に進む前の判断材料がそろっている。

## レビュー2: 要件レビュー

工程2の成果物を見て、最小構成、優先順位、後回しにする範囲を確認する。

確認事項:

- 非 AI 検索だけでも目的を満たす導線になっているか。
- PDF 関連の扱いが過剰または不足していないか。
- 追加調査の記録方法が実用的か。
- 実装前に技術検証が必要な要素はあるか。

終了条件:

- 試作または詳細設計に進む範囲が決まっている。

## 工程3: 試作または詳細設計

レビュー2で決めた範囲について、Web 画面、検索、情報登録、PDF 取り扱い、AI 秘書のいずれかを小さく試作または詳細設計する。実装に入る場合も、先に最小限の検証単位を決める。

成果物:

- 最小プロトタイプまたは詳細設計メモ
- 検索方式の候補と制約
- PDF 処理方式の候補と制約
- 実装する場合のタスク分解

完了条件:

- ユーザーが動くもの、または実装前の設計を確認できる。
- 主要リスクと後続タスクが分かる。

## レビュー3: 試作・設計レビュー

工程3の結果を確認し、本格実装に進むか、要件や設計を戻して直すかを判断する。

確認事項:

- `goal.md` の成功条件に近づいているか。
- 検索、参照、追加調査、PDF 取り扱いの中で不足している部分はどこか。
- AI 秘書が補助にとどまり、非 AI 検索を妨げていないか。
- 次に実装する最小単位は何か。

終了条件:

- 本格実装、再設計、追加調査のいずれに進むかが決まっている。

## 工程4: 実装・検証・整理

レビュー3後に決まった範囲を実装または具体化し、成果物を使える状態に近づける。工程中の判断は `issue_log/` に残し、worker ごとの成果物は `sub_artifact/NNN_slug/` に分ける。

成果物:

- 実装済み機能または確定仕様
- 操作・検証メモ
- 残課題と次アクション

完了条件:

- `goal.md` の成功条件と照合できる。
- 残課題が明示されている。
- ユーザーが成果物を利用、確認、または次の issue に分解できる。

## ユーザーへの確認事項

- 最初の次工程は、要件定義、画面原案、技術検証、プロトタイプのどれにするか。
- 扱いたい情報の種類と量はどの程度か。
- PDF スキャンは OCR まで必要か。
- AI 秘書は必須機能か、後回しにできる補助機能か。
- データの保存場所や公開範囲に制約はあるか。
